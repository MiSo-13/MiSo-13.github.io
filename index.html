문제

고객사별 점검 이력과 결과 데이터를 비교해 변화 내용을 파악하고, 이를 바탕으로 보고서 생성과 메일 발송까지 처리할 수 있는 기능이 필요했습니다.
하지만 기존 솔루션에는 데이터 비교와 보고서 생성 기능이 없어 운영팀이 반복적인 데이터 비교와 메일 발송 업무를 수작업으로 처리해야 했고,
이에 따라, 고객사 요청에 대응하는 시간도 지연될 수 밖에 없었습니다.

해결

추가 개발이 필요한 기능을 별도의 서비스로 분리하고, 공통 모듈을 정리해 중복 로직을 최소화했습니다.
또한 API Server와 Batch Server를 분리해 오래 걸리는 보고서 생성 작업을 비동기로 처리하고,
동적 스케줄링을 적용해 데이터 수집 · 비교 · 보고서 생성 · 메일 발송까지 한 흐름으로 자동화했습니다.

결과

기존 서비스에서 불가능했던 점검 이력 기반 데이터 비교와 보고서 생성 기능을 제공할 수 있게 되었고, 운영팀이 직접 보고서를 생성하고 관리할 수 있는 구조를 마련했습니다. 그 결과 고객사 대응 속도가 개선되었고, 약 20개 고객사를 대상으로 주 1회 데이터 수집, 비교, 보고서 생성, 메일 발송 일정을
자동화할 수 있었습니다.

기술적 인사이트

  • 사내 서비스와 연동되는 별도의 서비스를 구현하며, 서비스 분리와 연동 구조를 직접 다뤄볼 수 있었습니다.
    이 과정에서 정확한 도메인 분리가 확장성과 운영성 측면에서 훨씬 효과적이었습니다.
  • 오래 걸리거나 비동기로 진행되는 작업은 사용자 요청 흐름에서 분리하고,
    Batch Server(별도 서비스)에서 처리해야 응답성과 유지보수 효율을 함께 챙길 수 있었습니다.
  • 대용량 데이터를 다루는 기능에서는 저장, 비교, 보고서 생성의 전 과정에서 병목 지점을 구체적으로 확인해야 했고,
    서비스 연동 구조를 검토하고 적절한 디자인 패턴을 적용하는 과정이 성능 개선에 직접적인 도움이 됐습니다.
  • 실제 운영에서 사용하는 보고서와 메일 발송 기능은 기능 구현 자체도 중요하지만,
    운영 중 장애 대응(실패 알림 및 재시도 로직)과 같은 지속적인 최적화가 안정성에 더 큰 영향을 줬습니다.

자동화 파이프라인

  • Quartz를 사용해 사용자가 원하는 주기 및 일시에 동작 예약
  • Eureka / Gateway를 이용해 사내 다른 서비스와 연동하여 데이터 수집 및 저장
  • 이전 데이터와 비교하여 주간 보고서 생성
  • 비교한 데이터 기반으로 이메일 템플릿에 적용 및 각 고객사 별 정해진 일정에 메일 송부

트러블슈팅

  • Excel 보고서 생성 시 각 Cell 마다 Style을 적용하는 부분으로 인해 생성 지연

  • API 요청이나 보고서 생성 중 오류가 발생했을 때, 즉각적인 대응 불가능
    → 실패 시 관리자 메일로 오류 내용을 발송하고, 재시도가 가능하도록 구현하여 운영시 장애 대응 및 불편함 최소화

Java Spring Boot Spring Cloud Gateway Spring Quartz Kafka Redis Eureka JPA MyBatis MySQL

문제

점검 요청 저장과 Kafka 이벤트 발행 상태가 어긋나면서 조회 오류가 발생하거나 요청이 누락되는 문제가 있었습니다.
점검 결과 처리도 대용량 데이터를 Kafka body에 직접 담는 구조 때문에 메모리 이슈가 반복됐고,
이를 해결하기 위해 메모리 설정만 계속 늘리는 악순환이 있었습니다. 또한 예약 점검은 30초마다 DB를 조회하는 polling 방식이라 비효율적이었습니다.

해결

Outbox Pattern을 도입해 점검 요청 저장과 이벤트 발행을 하나의 트랜잭션으로 묶어 데이터 일관성과 장애 대응 가능성을 높였습니다.
점검 결과의 연동 방식도 Core 서비스가 고유한 Redis Key를 기준으로 요청과 결과를 주고받도록 전달 구조를 변경했습니다.
또한 Spring Quartz 기반으로 예약 점검이 정확한 시점에 실행되도록 개선했고, 사용자가 선택한 옵션에 맞게 자산과 포트별로 실제 필요한 점검 단계만 실행되도록 Plan 저장 구조를 조정했습니다.

결과

점검 요청 누락과 조회 오류 가능성을 줄였고, 실패하더라도 재시도할 수 있는 구조를 마련했습니다.
기존 엔진별로 각기 달랐던 결과 전달 흐름도 더 일관되고 단순하게 정리되면서 장애 대응 난이도와 메모리 이슈가 완화되며 유지보수성도 높아졌습니다.
또한 예약 점검 실행과 단계 생성 방식이 정리되면서 점검 진행 시간도 단축됐습니다.
그 결과 더 정밀한 점검 단계를 설계하고 운영할 수 있게 됐습니다.

기술적 인사이트

  • Outbox Pattern을 적용하면서 분리되어 있던 DB 저장과 이벤트 발행의 일관성을 유지하는 방식을 실무에 적용해볼 수 있었던 경험이 되었습니다.
  • Kafka 메모리 이슈를 해결하는 과정에서 Sequence Diagram으로 레거시 흐름을 다시 정리해보며,
    구조를 새로 그려보는 작업이 서비스 이해도와 개선 방향을 정리해 볼 수 있는 경험이 되었습니다.
  • Event 방식과 REST 방식이 함께 쓰이는 연동 환경을 경험하면서, MSA 환경을 더 깊이 이해하게 됐고, 헥사고날 방식에도 흥미가 생겨서 개인 프로젝트에 적용해볼 수 있었습니다.
  • 기존 레거시 코드 리팩터링뿐 아니라 DB 스키마를 PostgreSQL에서 MySQL로 마이그레이션하는 과정에서 ERD도 다시 그려보며, 코드와 스키마를 함께 다듬는 구조 개선 작업에 좋은 경험이였습니다.

Outbox Pattern 적용

  • @Transaction 블록에서 OutboxEntity 저장 및 OutboxEvent 생성
  • @TransactionalEventListener 을 이용해서, Transaction이 정상적으로 종료된 OutboxEvent를 받음
  • OutboxEntity에서 미리 정의된 eventCode와 payload 등을 조회해서 정해진 동작 수행
  • 만약 @Transaction 블록에서 실패시 @TransactionalEventListener가 받을 Event가 발행되지 않기 때문에 동작하지 않음

트러블슈팅

  • 점검을 수행하는 엔진마다 점검을 실행하는 이벤트 처리가 조금씩 다르고, Kafka에 지속적인 메모리 오류
    → 모든 엔진의 교환 방식을 Redis에 데이터를 저장하고 그 Key를 교환하는 방식으로 일관적으로 정리 및 적용
  • 서버의 장애 발생 등의 이유로 서버가 구동하고 있지 않을 때, 예약 점검이 실행되지 않는 문제
    → ApplicationRunner를 상속받는 Component를 이용해서 다시 scheduler를 활성화하고,
    특정 기준을 통해서 예약 점검을 바로 실행하거나 다음 실행 주기로 미루는 코드 구현
