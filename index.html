문제

점검 요청 저장과 Kafka 이벤트 발행 시점이 어긋나, Topic을 수신한 서비스가 아직 commit되지 않은 데이터를 조회하려다 오류가 발생했습니다.
점검 결과 처리도 대용량 데이터를 Kafka body에 직접 담는 구조라 메모리 이슈가 반복됐고, 엔진별 결과 전달 방식이 달라 장애 분석과 재처리가 어려웠습니다.

해결

Polling 기반 Outbox Pattern을 도입해 점검 요청 저장과 이벤트 발행의 정합성을 확보했습니다.
결과 전달은 Redis에 payload를 저장하고 Kafka로 Key만 전달하는 구조로 바꿨고, 실패 데이터는 운영용 API로 재처리할 수 있도록 멱등성과 Stack Trace 저장을 함께 설계했습니다.

결과

요청 유실·조회 불일치 문제를 테스트 환경 기준 30%에서 0건으로 해소했습니다.
Redis Key 전달 구조로 payload 부담을 줄여 관련 이슈를 5% 미만으로 낮췄고, 공통 workflow/state model 적용으로 신규 점검 단계 2개를 확장했습니다.

기술적 인사이트

  • Outbox Pattern을 적용하면서 DB 저장과 이벤트 발행의 일관성을 유지하는 방식을 실무에 적용해볼 수 있었습니다.
  • Kafka 메모리 이슈를 해결하는 과정에서 Sequence Diagram으로 레거시 흐름을 다시 정리해보며,
    구조를 새로 그려보는 작업이 서비스 이해도와 개선 방향을 정리하는 데 도움이 됐습니다.
  • Event 방식과 REST 방식이 함께 쓰이는 연동 환경을 경험하면서, MSA 환경을 더 깊이 이해하게 됐고, 헥사고날 방식에도 흥미가 생겨
    개인 프로젝트에 적용해볼 수 있었습니다.
  • 기존 레거시 코드 리팩터링뿐 아니라 DB 스키마를 PostgreSQL에서 MySQL로 마이그레이션하는 과정에서 ERD도 다시 그려보며,
    코드와 스키마를 함께 다듬는 구조 개선 작업을 경험했습니다.

Outbox Pattern 적용

  • @Transaction 블록에서 OutboxEntity 저장 및 OutboxEvent 생성
  • @TransactionalEventListener를 이용해서 Transaction이 정상적으로 종료된 OutboxEvent를 수신
  • OutboxEntity에서 미리 정의된 eventCode와 payload 등을 조회해서 정해진 동작 수행
  • 만약 @Transaction 블록에서 실패하면 Event가 발행되지 않기 때문에 동작하지 않음
  • 1분 주기로 동작하는 프로세스를 구현하여, 발행에 실패한 이벤트를 조회하여 재시도 실행

트러블슈팅

  • 점검을 수행하는 엔진마다 이벤트 처리가 조금씩 다르고, Kafka에 지속적인 메모리 오류 발생
    → 모든 엔진의 교환 방식을 Redis에 데이터를 저장하고 그 Key를 교환하는 방식으로 일관되게 정리 및 적용
  • 서버의 장애 등으로 서버가 구동하고 있지 않을 때 예약 점검이 실행되지 않는 문제
    → ApplicationRunner를 상속받는 Component를 이용해서 다시 scheduler를 활성화하고,
    특정 기준을 통해 예약 점검을 바로 실행하거나 다음 실행 주기로 미루는 코드 구현
