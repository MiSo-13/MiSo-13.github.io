회사 업무 · 1인 개발

프로젝트 소개

SooJi.Bee 결과 데이터를 수집하고 비교해, 정기 보고서 생성과 요약 메일 발송까지 자동화한 서비스입니다.

성과

20개 고객사를 대상으로 주 1~2회 자동화 운영이 가능해졌고, 보고서 생성 시간도 10분 이상에서 1분 미만으로 줄였습니다.

인원

1인 개발로 백엔드, 운영 화면, DB 스키마 설계를 맡았습니다.

핵심 기능

  • 사용자 설정 기반 동적 스케줄링
  • Snapshot 수집 및 데이터 비교
  • 주간 보고서 생성과 메일 발송 자동화
Java Spring Boot Spring Quartz JPA MyBatis MySQL

문제

고객사별 점검 결과를 주기적으로 수집하고 비교해서 분석할 수 있는 보고서와 메일 발송 기능이 필요했습니다. 기존 솔루션에는 데이터 비교와 보고서 생성 기능이 없었고, 반복 작업도 많아 자동화가 필요했습니다.

해결

별도 서비스로 분리하고 공통 모듈을 정리했으며, API Server와 Batch Server를 나눠 오래 걸리는 보고서 생성 작업을 비동기로 처리했습니다. 동적 스케줄링으로 수집, 비교, 보고서 생성, 메일 발송까지 자동화했습니다.

결과

기존 서비스에서 어려웠던 데이터 비교와 보고서 생성이 가능해졌고, 운영팀이 직접 보고서를 만들고 관리할 수 있어 고객사 대응 속도도 빨라졌습니다.

배운 점

오래 걸리는 작업은 사용자 요청 흐름에서 분리하고, 익숙한 기능도 병목 지점을 정확히 보면 크게 개선할 수 있다는 점을 다시 확인했습니다.

프로젝트 소개

Python / Django 기반 ASM 솔루션을 Java / Spring 구조로 다시 구성하며, 서비스 연동과 점검 실행 흐름을 고도화한 프로젝트입니다.

성과

Outbox와 CDC를 도입해 요청 유실 가능성을 줄였고, Redis 전달 구조로 대용량 전송 시 메모리 부담도 완화했습니다.

인원

BE 3명, FE 2명으로 진행한 프로젝트에서 백엔드를 담당했습니다.

핵심 기능

  • 레거시 기능의 Java / Spring 마이그레이션
  • Outbox Pattern과 CDC 기반 연동 구조 개선
  • 예약 점검 구조와 조회 API 개선
Java Spring Boot Kafka Redis JPA MySQL

문제

점검 요청 저장과 Kafka 발행 상태가 어긋날 수 있었고, 서비스별 저장소가 나뉘어 있어 연계 조회가 복잡했습니다. 대용량 점검 결과 전달 과정에서는 메모리 이슈가 있었고, 페이지네이션이 없는 조회 API도 사용성이 좋지 않았습니다.

해결

Outbox Pattern으로 저장과 발행의 경계를 분명히 하고, CDC로 필요한 조회 모델을 따로 만들었습니다. 대용량 결과는 Redis에 저장하고 Key만 전달하도록 바꿨으며, polling 대신 스케줄 기반 예약 점검 구조를 적용했습니다.

결과

요청 유실 가능성과 조회 오류를 줄였고, 연계 조회도 더 단순하게 정리할 수 있었습니다. 중복 점검이 실행되지 않도록 흐름을 바꾸면서 운영 구조도 더 명확해졌습니다.

배운 점

서비스가 분리된 환경에서는 이벤트 발행 자체보다도, 실패 복구 전략과 조회 모델 설계가 함께 중요하다는 점을 많이 배웠습니다.
