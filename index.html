Java Spring Boot JPA MyBatis MySQL React.js Jenkins

문제

고객사별 결과 데이터를 주기적으로 수집하고 비교해 보고서와 메일로 전달하는 과정이 반복적이었고, 기존 서비스만으로는 비교와 확인 업무를 충분히 지원하기 어려웠습니다. 보고서 생성 시간도 길어 사용자가 기다려야 했습니다.

해결

API Server와 Batch Server를 분리해 장시간 실행되는 작업을 배치로 이관했고, 사용자 설정 기반 동적 스케줄링으로 수집, 비교, 주간 상세·요약 보고서 생성, 메일 발송까지 자동화했습니다. 사내 타 서비스와의 API 연동도 추가했습니다.

결과

약 20개 고객사를 대상으로 주 1~2회 자동화 운영이 가능해졌고, 고객사별 최대 약 1,000개 자산 데이터를 처리했습니다. Excel 생성 병목을 개선해 10분 이상 걸리던 작업을 1분 미만으로 줄였습니다.

배운 점

사용자가 기다리는 요청 흐름 안에 장시간 작업을 두지 않는 것이 중요하다는 점, 그리고 익숙한 문서 생성 로직도 병목을 계측하면 큰 폭으로 개선할 수 있다는 점을 배웠습니다.

Java Spring Boot Kafka Redis JPA MyBatis MySQL

문제

DB 저장 상태와 Kafka 발행 상태가 어긋날 수 있었고, 서비스별로 데이터 저장소가 분리돼 있어 연계 조회가 복잡했습니다. 대용량 점검 결과 전송 과정에서는 메모리 이슈도 있었습니다.

해결

Outbox Pattern을 도입해 저장과 발행의 책임 경계를 분명히 했고, CDC로 필요한 조회 모델을 따로 구성했습니다. 또한 대용량 결과는 Redis에 저장하고 Key만 전달하도록 바꿨으며, 30초 polling 대신 스케줄 기반 예약 점검 구조를 구현했습니다.

결과

요청 유실 가능성을 줄이고, 서비스 간 연계 조회를 단순화했으며, 대용량 전송 과정의 메모리 부담을 완화했습니다. 중복 점검이 실행되지 않도록 흐름을 개편해 운영 안정성도 높였습니다.

배운 점

연동이 많은 시스템에서는 비동기 처리 자체보다 실패 복구 경로와 조회 모델까지 함께 설계해야 운영 안정성이 높아진다는 점을 배웠습니다.
