문제

점검 요청 저장과 Kafka 이벤트 발행 상태가 어긋나면서 조회 오류가 발생하거나 요청 누락 문제가 있었습니다.
점검 결과 처리도 대용량 데이터를 Kafka payload에 직접 담는 구조라 메모리 이슈와 네트워크 장애 대응 부담이 컸고,
서비스별 DB가 분리된 환경이라 연계 조회 구조도 복잡했습니다.

해결

Outbox Pattern을 도입해 점검 요청 저장과 이벤트 발행을 하나의 트랜잭션으로 묶어 데이터 일관성과 장애 대응 가능성을 높였습니다.
점검 결과의 연동 방식도 Core 서비스가 Redis Key를 기준으로 요청과 결과를 주고받도록 전달 구조를 변경했고, CDC를 적용해 서비스별로 분리된 데이터를 조회 가능한 형태로 동기화했습니다.
또한 기존 흐름을 표준화해 신규 점검 단계 2개를 안정적으로 추가할 수 있도록 구조를 정리했습니다.

결과

테스트 환경에서 조회 오류와 요청 유실을 0건으로 해소했고, Redis · Kafka 기반 전달 방식으로 이슈 발생 확률을 5% 미만으로 낮추며 장애 대응 시간을 80% 이상 단축했습니다.
또한 CDC 기반 조회 구조 정리로 연계 조회 속도를 40% 이상 개선했고, 신규 점검 단계도 안정적으로 확장할 수 있게 됐습니다.

기술적 인사이트

  • Outbox Pattern을 적용하면서 DB 저장과 이벤트 발행의 일관성을 유지하는 방식을 실무에 적용해볼 수 있었습니다.
  • Kafka 메모리 이슈를 해결하는 과정에서 Sequence Diagram으로 레거시 흐름을 다시 정리해보며,
    구조를 새로 그려보는 작업이 서비스 이해도와 개선 방향을 정리하는 데 도움이 됐습니다.
  • Event 방식과 REST 방식이 함께 쓰이는 연동 환경을 경험하면서, MSA 환경을 더 깊이 이해하게 됐고, 헥사고날 방식에도 흥미가 생겨
    개인 프로젝트에 적용해볼 수 있었습니다.
  • 기존 레거시 코드 리팩터링뿐 아니라 DB 스키마를 PostgreSQL에서 MySQL로 마이그레이션하는 과정에서 ERD도 다시 그려보며,
    코드와 스키마를 함께 다듬는 구조 개선 작업을 경험했습니다.

Outbox Pattern 적용

  • @Transaction 블록에서 OutboxEntity 저장 및 OutboxEvent 생성
  • @TransactionalEventListener를 이용해서 Transaction이 정상적으로 종료된 OutboxEvent를 수신
  • OutboxEntity에서 미리 정의된 eventCode와 payload 등을 조회해서 정해진 동작 수행
  • 만약 @Transaction 블록에서 실패하면 Event가 발행되지 않기 때문에 동작하지 않음
  • 1분 주기로 동작하는 프로세스를 구현하여, 발행에 실패한 이벤트를 조회하여 재시도 실행

트러블슈팅

  • 점검을 수행하는 엔진마다 이벤트 처리가 조금씩 다르고, Kafka에 지속적인 메모리 오류 발생
    → 모든 엔진의 교환 방식을 Redis에 데이터를 저장하고 그 Key를 교환하는 방식으로 일관되게 정리 및 적용
  • 서버의 장애 등으로 서버가 구동하고 있지 않을 때 예약 점검이 실행되지 않는 문제
    → ApplicationRunner를 상속받는 Component를 이용해서 다시 scheduler를 활성화하고,
    특정 기준을 통해 예약 점검을 바로 실행하거나 다음 실행 주기로 미루는 코드 구현

문제

고객사별 점검 이력과 결과 데이터를 비교해 변화 내용을 파악하고, 이를 바탕으로 보고서 생성과 메일 발송까지 처리할 수 있는 기능이 필요했습니다.
하지만 기존 솔루션에는 데이터 비교와 보고서 생성 기능이 제한적이어서 운영팀이 반복적인 데이터 비교와 메일 발송 업무를 수작업으로 처리해야 했고,
이에 따라 고객사 요청에 대응하는 시간도 지연될 수밖에 없었습니다.

해결

추가 개발이 필요한 기능을 멀티 모듈 서비스로 분리하고, 공통 모듈을 정리해 중복 로직을 최소화했습니다.
또한 API Server와 Batch Server를 분리해 오래 걸리는 보고서 생성 작업을 비동기 처리하고,
동적 스케줄링을 적용해 데이터 수집 · 비교 · 보고서 생성 · 메일 발송까지 자동화했습니다.

결과

기존 서비스에서 제한적이었던 데이터 비교와 보고서 생성을 자동화했고, 운영팀이 직접 보고서를 생성하고 관리할 수 있는 구조를 마련했습니다.
16개 고객사를 대상으로 주 1회 자동화 일정을 운영하며, 누적 1,000건 이상의 보고서를 생성했고 요청당 최대 1,000개 자산과 관련 서브도메인, 위협, 포트 데이터도 안정적으로 처리할 수 있게 됐습니다.

기술적 인사이트

  • 사내 MSA 구조에서 별도 서비스를 구현하며, 서비스 분리와 연동 구조를 직접 다뤄볼 수 있었습니다.
    이 과정에서 정확한 도메인 분리와 확장성, 운영성을 고려해 성공적으로 출시할 수 있었습니다.
  • 오래 걸리거나 비동기로 진행되는 작업은 요청 흐름에서 분리하고,
    Batch Server에서 처리해서 응답성과 유지보수 효율을 함께 챙길 수 있었습니다.
  • 대용량 데이터를 다루는 기능에서는 저장·비교·보고서 생성 전 과정에서 병목 지점을 구체적으로 확인해야 했고,
    연동 구조를 검토하고 적절한 디자인 패턴을 적용하는 과정이 성능 개선에 직접적인 도움이 됐습니다.
  • 실제 운영에서 사용하는 보고서와 메일 발송 기능은 기능 구현 자체도 중요하지만,
    장애 대응(실패 알림 및 재시도 로직)과 같은 지속적인 최적화가 안정성에 더 큰 영향을 준다는 것을 느꼈습니다.

주요 자동화 파이프라인

  • Quartz를 사용해 사용자가 원하는 주기 및 일시에 동작 예약
  • Eureka / Gateway를 이용한 REST 통신으로 사내 다른 서비스와 연동하여 데이터 수집 및 저장
  • 이전에 수집된 데이터와 비교하여 주간 보고서 생성
  • 비교한 데이터를 이메일 템플릿에 적용하고, 각 고객사별 정해진 일정에 메일 송부

트러블슈팅

  • Excel 보고서 생성 시 15분 정도가 소요되는 현상 발견
    → Cell Style 처리에서 문제가 발생하는 것을 파악하고,
    Singleton Pattern을 적용해 동일한 Style이 반복 생성되지 않도록 구조를 개선하여 15분 수준이던 작업을 2분 미만으로 단축
  • API 요청이나 보고서 생성 중 오류가 발생했을 때 즉각적인 대응이 어려움
    → 실패 시 관리자 메일로 오류 내용을 발송하고, 재시도가 가능하도록 구현해 운영 중 장애 대응과 불편함 최소화
  • 대용량 데이터 조회 및 비교 시 지연 발생
    → INDEX 활용 및 캐싱으로 조회 시간 단축 (30초 → 2초)
    → DB 조회보다 코드 집계가 더 빠르다고 판단되는 부분은 리팩터링

회고

설계 초기에 요구사항 규모와 복잡도를 고려했을 때, Spring Batch보다 Quartz의 단순한 메서드 호출 방식이 적절하다고 판단했습니다.
실제로 오류 트래킹, 실패 알림, 재시도 로직 등을 직접 구성해 운영할 수 있었지만, 돌아보면 배치 실행 단위의 상태 관리와 재처리 흐름, 모니터링을
더 표준화된 방식으로 가져가기 위해 Spring Batch를 적용했다면 운영 안정성을 더 높일 수 있었겠다는 생각을 하게 되었습니다.
